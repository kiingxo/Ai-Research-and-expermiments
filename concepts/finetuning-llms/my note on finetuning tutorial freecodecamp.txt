Full Precision vs Half Precision
	•	Full precision: Typically uses 32-bit floating point (FP32)
	•	Half precision: Uses 16-bit (FP16 or BF16), reducing memory and compute usage

⸻

Revisiting Quantization

Quantization is the process of converting a model from a higher-precision format to a lower-precision format.

Goal:
	•	Reduce memory footprint
	•	Speed up inference
	•	Keep model accuracy close to original

Example:
	•	Original model trained in FP32 (32-bit float)
	•	Can be quantized to:
	•	FP16 (16-bit float)
	•	INT8 (8-bit integer)
	•	Even 4-bit (used in QLoRA)

⸻

Why Quantization Matters During Fine-tuning

When fine-tuning LLMs (like LLaMA, Mistral, or MPT) on laptops or consumer GPUs, full-precision models may not fit in memory.

Example Use Case:
Fine-tuning a 7B LLaMA model:
	•	FP32 version needs ~60–80GB VRAM
	•	4-bit quantized version (QLoRA) fits into 16–24GB VRAM

This makes it possible to:
	•	Run on one GPU (e.g., RTX 3090 or Apple M2 with 32–64GB RAM)
	•	Use QLoRA for fine-tuning with low memory but good accuracy

⸻

Types of Quantization
	1.	Post-Training Quantization: Applied after model training.
	2.	Quantization-Aware Training (QAT): Model is trained with quantization in mind.

Precision Levels
	•	FP32: Full precision (baseline)
	•	FP16 / BF16: Half precision
	•	INT8: 8-bit integer quantization
	•	4-bit: Aggressive compression (used in QLoRA)

Symmetric vs Asymmetric Quantization

Quantization is about converting float values (e.g. 32-bit floats) into smaller integer representations (like 8-bit), to reduce model size and speed up inference.

There are two main strategies:

⸻

Symmetric Quantization
• The float values are assumed to be centered around 0 (like from -1.0 to +1.0).
• The zero-point is always 0.
• Uses the same scale factor across the tensor.
• Commonly used for model weights.

Formula:
	quantized = round(float_value / scale)

Example:
	If your float values range from -1.0 to +1.0, and scale = 1.0 / 127, then:
	- float -1.0 → int -127
	- float  0.0 → int   0
	- float +1.0 → int +127

⸻

Asymmetric Quantization
• The float values are not centered around 0 — may be all positive (like from 0 to 6).
• A non-zero zero-point is used to shift the range.
• Each tensor (or channel) can have a different scale and zero-point.
• Often used for activations (e.g. outputs of ReLU).

Formula:
	quantized = round((float_value / scale) + zero_point)

Example:
	If float values range from 0.0 to 6.0, and scale = 6.0 / 255, zero_point = 0:
	- float 0.0 → int 0
	- float 3.0 → int 127
	- float 6.0 → int 255

How Quantization Works: Float to Int Mapping

To understand how a float (like FP32) is converted to an integer (like INT8 or 4-bit), you need two things:
1. A **scale** — tells how much to stretch or shrink the float range
2. A **zero-point** (only for asymmetric) — tells where 0 in float maps in int space

General formulas:
• **Symmetric**: 
    quantized = round(float_value / scale)
    dequantized = quantized * scale

• **Asymmetric**: 
    quantized = round((float_value / scale) + zero_point)
    dequantized = (quantized - zero_point) * scale

Example – Symmetric:
Let’s say your float values range from -1.0 to 1.0. You want to quantize to INT8 (range -127 to 127).

- scale = (max(abs(min), abs(max))) / 127 = 1.0 / 127 ≈ 0.00787
- float = 0.5
- quantized = round(0.5 / 0.00787) ≈ 64

Example – Asymmetric:
Float values range from 0 to 6.0, quantized to UINT8 (0 to 255):

- scale = (6.0 - 0.0) / 255 ≈ 0.0235
- zero_point = 0
- float = 3.0
- quantized = round((3.0 / 0.0235) + 0) ≈ 128

Dequantization lets you approximate the original value from the quantized int:
- dequantized = 128 * 0.0235 = 3.008

Why This Matters:
- Quantization is lossy (you can’t perfectly recover the original float)
- The goal is to minimize error and still let the model perform well
