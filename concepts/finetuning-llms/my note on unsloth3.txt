
# Fine-tuning LLMs — Beyond the Basics  

###  Techniques & Variations
- **Full Fine-tuning**  
  Update *all* parameters. Powerful but requires huge compute + risk of catastrophic forgetting. Rarely used outside research.  
- **Parameter-Efficient Fine-Tuning (PEFT)**  
  Only train a *small subset* of parameters:  
  - **LoRA**: Injects low-rank adapters into attention layers. Saves VRAM & compute.  
  - **QLoRA**: Quantizes base model (4-bit) + applies LoRA → big models can fit on a single GPU.  
  - **Adapters / Prefix-Tuning**: Extra tokens or layers trained instead of touching the full model.  
- **Instruction-tuning**  
  Curating examples of prompts → responses to align model with tasks (like Alpaca, Dolly).  
- **RLHF / DPO (Direct Preference Optimization)**  
  Train models to *prefer* outputs by comparing pairs rather than predicting single labels.  

---

###  Things to Consider Before/During Fine-Tuning
1. **Data Quality > Quantity**  
   Garbage in → garbage out. Models are very sensitive to annotation mistakes, formatting issues, or inconsistent style.  
2. **Domain Gap**  
   If your fine-tune dataset is *too different* from pretraining data, model may underperform.  
   → Consider *continued pretraining* on domain corpus before instruction fine-tuning.  
3. **Overfitting vs Underfitting**  
   - Too much training = memorization.  
   - Too little = no real learning (just base model responses).  
   Monitoring validation loss is key.  
4. **Tokenization Issues**  
   Ensure your dataset is properly tokenized with the *same tokenizer* as the base model. Otherwise, embeddings misalign.  
5. **Learning Rate Schedules**  
   Start small (1e-5 to 2e-5 for LLMs) → warmup → decay. A poor LR can either freeze learning or destabilize training.  
6. **Batch Size & Gradient Accumulation**  
   Bigger batch sizes give smoother gradients, but memory is limiting → use gradient accumulation to simulate larger batches.  
7. **Evaluation & Metrics**  
   - For classification: accuracy, F1.  
   - For generation: BLEU, ROUGE, perplexity, human eval.  
   - Sometimes perplexity drops but quality *looks worse* — always sanity check outputs.  

---

###  Losses in LLM Fine-tuning
- **Cross-Entropy Loss (Standard)**  
  The model predicts token probabilities. Loss penalizes deviation from the gold sequence.  
- **KL Divergence**  
  Useful in preference optimization (e.g., DPO) to keep fine-tuned model close to base model.  
- **Reward Models (RLHF)**  
  Train a separate reward function → fine-tune model with PPO (policy gradient) to maximize reward.  
- **Custom Losses**  
  - Contrastive losses (for embedding alignment).  
  - Span losses (for extractive tasks like QA).  

**Why Loss Matters**:  
Loss is essentially the “score” that guides optimization. The optimizer tries to minimize this score by adjusting parameters.  
- Low loss → predictions are close to target.  
- High loss → predictions are wrong.  
Monitoring the *training vs validation loss* shows if the model is learning properly or overfitting.

---

###  Optimization Details
1. **Optimizer Choices**
   - **AdamW** (default for LLM fine-tuning).  
   - **Lion / Sophia optimizers** sometimes improve convergence.  
2. **Gradient Clipping**  
   Prevents exploding gradients in large models.  
3. **Mixed Precision (FP16 / BF16)**  
   Speeds up training, saves memory — standard in Unsloth/PEFT setups.  
4. **Regularization**
   - Weight decay (0.01 typical).  
   - Dropout in adapters.  

---

###  Advanced Tricks
- **Curriculum Fine-tuning**  
  Start with easy examples → gradually increase difficulty.  
- **Data Augmentation**  
  Rephrase prompts, paraphrase responses → prevents overfitting.  
- **Freezing Layers**  
  Freeze lower transformer blocks → only adapt higher layers → saves compute.  
- **Checkpoint Averaging / SFT + DPO hybrid**  
  Combine multiple fine-tuned runs or blend supervised with preference training.  

---

###  Practical Fine-tuning Checklist
 Prepare clean dataset (instruction–response pairs, consistent format)  
 Choose PEFT method (LoRA, QLoRA) for efficiency  
 Set learning rate (1e-5 range), warmup steps, and optimizer (AdamW)  
 Use gradient accumulation if GPU memory is small  
 Track training loss + validation loss (watch for overfitting)  
 Evaluate outputs qualitatively (don’t trust loss alone)  
 Save checkpoints regularly  
 Optionally: continue with DPO or RLHF if you want preference alignment  

---

 **Summary**:  
Fine-tuning isn’t just “run Unsloth and get results.” The *real skill* is balancing data quality, training stability, loss monitoring, and evaluation. Most breakthroughs in practice come from **curating better datasets + using PEFT correctly**, not just raw compute.
